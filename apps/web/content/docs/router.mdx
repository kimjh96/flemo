`<Router>` is the root container. It picks which `<Route>` to render based on the URL.

## The pieces

```tsx
<Router>
  <Route path="/" element={<Home />} />
  <Route path="/posts/:slug" element={<Post />} />
</Router>
```

| Element    | Job                                          |
| ---------- | -------------------------------------------- |
| `<Router>` | Sets up history, transitions, and decorators |
| `<Route>`  | Maps a `path` (or paths) to an `element`     |

## Path patterns

flemo uses [path-to-regexp](https://github.com/pillarjs/path-to-regexp) v8 for matching.

```ts
"/"; // exact
"/posts/:slug"; // a single param
"/users/:id/posts/:p"; // multiple params
"/files/*splat"; // wildcard (note the `*` prefix on the param)
```

You can also pass an array to share one component across paths:

```tsx
<Route path={["/", "/home"]} element={<Home />} />
```

## Type-safe routes

Augment `RegisterRoute` and `navigate.push`, `useParams`, etc. all type-check against it. You can
declare every route in one central file, or split the declaration across page files ‚Äî TypeScript
merges all the augmentations, so either approach works. Most projects keep it at the bottom of
the file where `<Router>` is mounted.

```ts title="App.tsx"
declare module "@flemo/react" {
  interface RegisterRoute {
    "/": undefined;
    "/posts/:slug": { slug: string };
    "/users/:id": { id: string };
  }
}
```

Routes without params map to `undefined`. Routes with params use the inferred shape.

```tsx
navigate.push("/posts/:slug", { slug: "hello" }); // ‚úÖ
navigate.push("/posts/:slug", { id: "1" }); // ‚ùå TS error
navigate.push("/unknown"); // ‚ùå TS error
```

## Router options

```tsx
<Router initPath="/" defaultTransitionName="cupertino" transitions={[]} decorators={[]}>
  {/* routes */}
</Router>
```

| Prop                    | Default       | What it does                                                      |
| ----------------------- | ------------- | ----------------------------------------------------------------- |
| `initPath`              | `"/"`         | Path used during SSR before `window.location` is available        |
| `defaultTransitionName` | `"cupertino"` | Transition used when a `navigate.push` doesn't specify one        |
| `transitions`           | `[]`          | Custom transitions to register (see [Transitions](./transitions)) |
| `decorators`            | `[]`          | Custom decorators (e.g., overlays) to register                    |

## Server-side rendering

flemo is a **client-side SPA router** ‚Äî it drives `window.history` and takes over navigation once
it mounts. `<Router>` can still render on the server for the first paint, but the server has no
`window.location`, so you tell it which route to render first with `initPath`.

- **Server**: renders the screen matching `initPath`. Defaults to `"/"`.
- **Client**: on mount, reads `window.location.pathname` and takes over from there.

Pass the request path through `initPath` and the first paint matches the URL ‚Äî no hydration
mismatch. This works with any setup that renders React to a string and lets flemo own routing
from there:

```tsx
// On the server, hand the incoming request path to initPath.
<Router initPath={requestPath}>{/* routes */}</Router>
```

Pure-SPA setups (Vite, etc.) don't need `initPath` at all ‚Äî the client reads
`window.location.pathname` on its own.

<Callout type="warn">
  Because flemo owns client-side history (it pushes to `window.history` and listens for `popstate`),
  it **does not compose with a host framework that also owns routing** ‚Äî notably the **Next.js App
  Router**. The two routers fight over navigation and the URL, so flemo can't act as the router for
  a Next.js app's pages. Use flemo in a pure SPA, or mount it as a self-contained, client-only
  island that doesn't share routing with the host framework.
</Callout>

## Reading router state from outside the Router

Inside the Router tree ‚Äî any `<Route>` element and the components it renders ‚Äî navigation just
works: `useNavigate`, `useScreen`, and the store hooks below all read the active Router's state,
no setup required.

The stores are **request-scoped**: `<Router>` creates one bundle per mount and shares it through
context, so concurrent SSR requests stay isolated. The catch: a component rendered _outside_ the
Router ‚Äî a devtools panel, an inspector beside the screen frame ‚Äî has no context to read, and the
store hooks throw.

Wrap the Router and its outside siblings in `<RouterScopeProvider>` to share one bundle. The Router
adopts the hosted bundle instead of creating its own, so the sibling reads ‚Äî and can drive ‚Äî the
same state.

```tsx
import { Router, Route, RouterScopeProvider, useHistoryStore, useNavigate } from "@flemo/react";

function Inspector() {
  // Selector read ‚Äî re-renders when the stack changes, like a zustand hook.
  const depth = useHistoryStore((state) => state.histories.length);
  const navigate = useNavigate(); // drives the Router rendered beside it

  return (
    <aside>
      <p>stack depth: {depth}</p>
      <button onClick={() => navigate.pop()}>back</button>
    </aside>
  );
}

<RouterScopeProvider>
  <Router>
    <Route path="/" element={<Home />} />
    <Route path="/posts/:slug" element={<Post />} />
  </Router>
  <Inspector /> {/* outside the Router, same stores */}
</RouterScopeProvider>;
```

Every store hook reads the active Router's request-scoped stores:

| Hook                     | Reads                                                         |
| ------------------------ | ------------------------------------------------------------- |
| `useHistoryStore(fn)`    | The history stack (`index`, `histories`) and its mutators     |
| `useNavigateStore(fn)`   | Navigation status (`IDLE` / `PUSHING` / `POPPING` / ‚Ä¶)        |
| `useTransitionStore(fn)` | The default transition name                                   |
| `useScreenStore(fn)`     | Drag / replace status and the shared-bar registry             |
| `useStores()`            | All four store APIs for imperative `getState()` / `subscribe` |

The four selector hooks take a selector and re-render on change. `useStores()` returns the raw
store APIs for event handlers and effects that run outside React render.

<Callout>
  `<RouterScopeProvider>` is **opt-in** ‚Äî reach for it only when something outside the `<Router>`
  reads or drives its state. A normal app renders `<Router>` on its own and never needs it. The
  server snapshot from [SSR](#server-side-rendering) applies to the default path only; a hosted
  scope seeds on the client, so use it for client-side devtools layouts.
</Callout>
